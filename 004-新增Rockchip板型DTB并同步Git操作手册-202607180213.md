# 新增 Rockchip 板型 DTB 并同步 Git 操作手册

本文记录如何在本仓库中加入一个新的 Rockchip 板子 dtb，并让 GitHub Actions 构建页面出现这个新板子的选项。

当前方案适用于：上游 ImmortalWrt ImageBuilder 没有你的板型 profile，但你的板子可以借用某个现有 RK3399 profile 的启动/分区/打包流程，例如目前使用 `friendlyarm_nanopc-t4` 作为打包入口。

## 一、准备 dtb 文件

把新板子的 dtb 放到：

```text
rockchip/dtb/
```

文件名建议使用内核设备树命名风格：

```text
rk3399-板子名.dtb
```

例如：

```text
rockchip/dtb/rk3399-dg3399.dtb
rockchip/dtb/rk3399-boocax.dtb
```

## 二、修改 GitHub Actions 板型选项

需要同时修改 24.10 和 25.12 两个 Rockchip workflow：

```text
.github/workflows/build-rockchip-immortalWrt-24.10.x.yml
.github/workflows/build-rockchip-25.12.x.yml
```

找到 `profile` 的 `options` 列表，加入新板型名称。

示例：

```yaml
options:
  - cyber_cyber3588-aib
  - dg3399
  - boocax
  - yourboard
  - ezpro_mrkaio-m68s
```

这里的 `yourboard` 就是 GitHub Actions 页面中显示和选择的板型名。

## 三、修改 build24.sh 和 build25.sh

需要同时修改：

```text
rockchip/build24.sh
rockchip/build25.sh
```

找到函数：

```bash
prepare_custom_rockchip_board() {
```

在 `case "$PROFILE" in` 中新增一个分支。

示例：

```bash
yourboard)
    CUSTOM_BOARD_NAME="yourboard"
    CUSTOM_BOARD_MODEL="YourBoard RK3399"
    CUSTOM_DTB="/home/build/immortalwrt/custom-dtb/rk3399-yourboard.dtb"
    IMAGEBUILDER_PROFILE="friendlyarm_nanopc-t4"
    CUSTOM_KERNEL_PIPE="kernel-bin | lzma | fit lzma $CUSTOM_DTB"
    CUSTOM_REMOVE_PACKAGES="-kmod-brcmfmac -brcmfmac-firmware-4356-sdio -brcmfmac-nvram-4356-sdio"
    ;;
```

字段说明：

- `CUSTOM_BOARD_NAME`：内部板型名，建议和 workflow 选项一致。
- `CUSTOM_BOARD_MODEL`：进入 ImmortalWrt 后显示的设备型号。
- `CUSTOM_DTB`：容器内部的 dtb 路径，文件名必须和 `rockchip/dtb/` 里的 dtb 对应。
- `IMAGEBUILDER_PROFILE`：借用的上游 ImageBuilder profile。目前 RK3399 自定义板使用 `friendlyarm_nanopc-t4`。
- `CUSTOM_KERNEL_PIPE`：关键项，让 FIT 镜像直接使用你的自定义 dtb。
- `CUSTOM_REMOVE_PACKAGES`：可选项，用于移除借用 profile 默认带入但你的板子不需要的包。

## 四、设备型号显示逻辑

自定义板型构建时，脚本会生成：

```text
/etc/init.d/custom-board-model
/etc/rc.d/S05custom-board-model
```

开机后它会写入：

```text
/tmp/sysinfo/model
/tmp/sysinfo/board_name
```

LuCI 和 `ubus call system board` 通常会读取这些信息，因此可以把界面里的设备型号从 `NanoPC-T4` 改成你的自定义型号。

注意：U-Boot 阶段显示的 `Model: FriendlyElec NanoPC-T4` 不属于 ImmortalWrt 用户态，通常来自 U-Boot 自己携带的设备树或环境，本文方案不修改它。

## 五、日志中的常见提示如何判断

以下提示通常不是致命问题：

```text
No find bl30.bin
No find bl32.bin
WARNING: No OPTEE provided by BL2 boot loader
ERROR: Error initializing runtime service opteed_fast
No EFI system partition
EFI boot manager: Cannot load any image
Malformed early option 'earlycon'
block: unable to load configuration (fstab: Entry not found)
overlay filesystem has not been fully initialized yet
```

这些多半来自 Rockchip 启动链、U-Boot 扫描 EFI、OpenWrt 首次初始化 overlay 或默认配置缺失。只要后面出现：

```text
VFS: Mounted root
procd: - init -
Please press Enter to activate this console.
br-lan: port 1(eth0) entered forwarding state
```

说明系统已经正常进入用户态。

以下提示需要结合硬件判断：

```text
rockchip-pcie ... PCIe link training gen1 timeout
```

如果你的板子没有接 PCIe 设备，通常可以忽略；如果需要 PCIe，则要回到 dts/dtb 中检查 PCIe 电源、复位 GPIO、时钟和 `status`。

```text
brcmfmac ... friendlyarm,nanopc-t4.bin failed
```

这是借用 NanoPC-T4 profile 带入了 Broadcom SDIO WiFi 驱动，但你的板子没有对应 WiFi 时产生的提示。当前脚本已对自定义板型移除 NanoPC-T4 默认 Broadcom WiFi 包。

## 六、提交并同步到 GitHub

修改完成后，在仓库根目录执行：

```powershell
git status -sb
git add .github/workflows/build-rockchip-immortalWrt-24.10.x.yml .github/workflows/build-rockchip-25.12.x.yml rockchip/build24.sh rockchip/build25.sh rockchip/dtb/rk3399-yourboard.dtb
git commit -m "Add yourboard Rockchip custom profile"
git push origin master
```

如果你还新增或修改了说明文档，也一起加入 `git add`。

确认远端地址：

```powershell
git remote -v
```

本仓库中 `origin` 指向自己的 fork：

```text
https://github.com/jiangmo/ImmortalWrt-ImageBuilder.git
```

推送到 `origin/master` 只会更新自己的 fork，不会影响 ImmortalWrt 上游仓库。

## 七、在线编译

推送完成后，打开 GitHub Actions。

选择对应 workflow：

```text
Build 24.10.x Rockchip
```

或：

```text
Build 25.12.x Rockchip
```

在 `profile` 下拉框中选择你新增的板型，例如：

```text
yourboard
```

然后手动触发构建。

## 八、启动后验证

串口日志中确认系统启动：

```text
VFS: Mounted root
procd: - init -
Please press Enter to activate this console.
```

进入系统后查看型号：

```sh
cat /tmp/sysinfo/model
cat /tmp/sysinfo/board_name
ubus call system board
```

查看网卡：

```sh
ip addr
logread | grep -i eth
```

如果网络正常，`br-lan` 应该有管理地址，`eth0` 应该加入 `br-lan`。

## 九、排错重点

如果启动后仍显示 NanoPC-T4：

1. 确认 Actions 中选择的是新板型，而不是 `friendlyarm_nanopc-t4`。
2. 确认 `rockchip/build24.sh` 和 `rockchip/build25.sh` 中都新增了对应 `case` 分支。
3. 确认 `CUSTOM_BOARD_MODEL` 已设置。
4. 进入系统后执行：

```sh
/etc/init.d/custom-board-model start
cat /tmp/sysinfo/model
```

如果设备树没有生效：

1. 检查 GitHub Actions 日志中是否打印：

```text
自定义设备树大小
自定义 KERNEL 打包管线
```

2. 确认 `CUSTOM_DTB` 文件名与 `rockchip/dtb/` 下实际文件名完全一致。
3. 确认 `CUSTOM_KERNEL_PIPE` 形如：

```text
kernel-bin | lzma | fit lzma /home/build/immortalwrt/custom-dtb/rk3399-yourboard.dtb
```

如果 PCIe、USB、WiFi、LED、按键异常，优先检查 dtb 本身。这类问题不是 ImageBuilder 脚本能完全修复的，通常需要回到 dts 修改硬件节点。
