# 修正自定义设备树 FIT 打包方式

## 问题现象

串口启动日志显示系统已经启动到 OpenWrt 用户态：

```text
procd: - init -
Please press Enter to activate this console.
br-lan: port 1(eth0) entered forwarding state
```

但 FIT 中加载的设备树仍然是原始 NanoPC-T4：

```text
Description: ARM64 OpenWrt friendlyarm_nanopc-t4 device tree blob
Data Size: 62842 Bytes
Machine model: FriendlyElec NanoPC-T4
```

本仓库中的 `rk3399-dg3399.dtb` 大小为 82421 bytes，`rk3399-boocax.dtb` 大小为 87867 bytes，因此上一次“查找并覆盖 `rk3399-nanopc-t4.dtb`”的方式没有进入最终 FIT 镜像。

## 修正方式

对 `dg3399` 和 `boocax` 两个自定义板型，不再尝试覆盖 ImageBuilder 内部已有的 `rk3399-nanopc-t4.dtb` 文件。

现在脚本会直接覆盖 ImageBuilder 的 `KERNEL` 打包管线：

```text
kernel-bin | lzma | fit lzma /home/build/immortalwrt/custom-dtb/rk3399-dg3399.dtb
kernel-bin | lzma | fit lzma /home/build/immortalwrt/custom-dtb/rk3399-boocax.dtb
```

这样 `fit lzma` 会直接使用仓库中挂载进去的自定义 dtb 作为 FDT 输入。

## 验证点

重新在线编译并启动后，串口日志中的 FIT FDT 大小应变化为对应 dtb 的大小：

```text
dg3399: 82421 Bytes
boocax: 87867 Bytes
```

内核启动后的 `Machine model` 也应来自自定义设备树，而不是 `FriendlyElec NanoPC-T4`。

## 注意

本次没有执行本地编译动作，编译仍由 GitHub Actions 手动触发。
